01 / Introducción

¿Qué son WhatsApp Flows?

WhatsApp Flows son experiencias interactivas nativas que se renderizan dentro del chat de WhatsApp, sin redirigir al usuario a un navegador o app externa. Funcionan como formularios multi-pantalla con lógica, validaciones y conexión a sistemas en tiempo real.

📋

Formularios Ricos

Campos de texto, dropdowns, radio buttons, checkboxes, date pickers, sliders y más.

📅

Agendamiento

Selector de fecha nativo con fechas bloqueadas, rango mínimo/máximo y slots disponibles.

🔗

Integración con APIs

Conexión en tiempo real con tus sistemas: CRM, ERP, bases de datos, calendarios.

🔐

Encriptación E2E

Los datos sensibles se envían encriptados con RSA-2048 + AES-128-GCM.

📊

Generación de Leads

Captura datos estructurados directamente en el chat, sin formularios web externos.

⚡

Sin Apps Externas

Toda la experiencia ocurre dentro de WhatsApp, reduciendo la fricción del usuario.

Tipos de Flows

02 / Requisitos

Requisitos para Implementar

Requisitos de Cuenta

  • ✓
    WhatsApp Business Account (WABA) aprobada — Creada y aprobada dentro de Meta Business Manager. No es la app normal de WhatsApp Business.
  • ✓
    Número de teléfono verificado — Un número dedicado registrado en la WABA. El número no puede estar en uso en otro dispositivo.
  • ✓
    Verificación de negocio en Meta — Meta debe verificar tu empresa (puede tomar 1–5 días hábiles). Se requiere documentación legal.
  • ✓
    Acceso a Meta Cloud API o BSP — Puedes acceder directamente vía Meta Cloud API (gratis) o a través de un Business Solution Provider (BSP) como Twilio, Infobip, 360dialog, Vonage.
  • ✓
    App de Meta para Desarrolladores — Para obtener tokens de acceso, configurar webhooks y usar la API programáticamente.

Requisitos Técnicos (Flows Dinámicos)

  • ✓
    Servidor HTTPS público — Tu endpoint de data exchange debe tener SSL/TLS válido (no self-signed). Puerto 443.
  • ✓
    Par de llaves RSA-2048 — Para encriptar/desencriptar el intercambio de datos entre WhatsApp y tu backend.
  • ✓
    Tiempo de respuesta < 10 segundos — Los endpoints deben responder en menos de 10s o el Flow mostrará error.
  • ✓
    Respuesta HTTP 200 con JSON válido — Cualquier código diferente causa error en el Flow.
ℹ️

Tiempo estimado de setup inicial: Para una empresa nueva sin WABA, el proceso completo (verificación, configuración, primer Flow funcionando) toma entre 3 y 10 días hábiles, principalmente por los tiempos de aprobación de Meta.

Proceso de Alta Paso a Paso

  • 1
    Crear cuenta en Meta Business Manager business.facebook.com — Registra tu empresa con nombre legal, dirección y datos fiscales.
  • 2
    Verificar el negocio en Meta Subir documentación (RUC, acta constitutiva, etc.). Tiempo: 1–5 días hábiles. Algunos BSPs agilizán este proceso.
  • 3
    Crear la WABA y registrar el número Dentro del Business Manager. El número necesita poder recibir un SMS o llamada de verificación.
  • 4
    Crear App en Meta for Developers developers.facebook.com — Tipo "Business". Vincular la WABA. Obtener tokens de sistema.
  • 5
    Configurar Webhook URL pública HTTPS donde Meta enviará eventos (mensajes recibidos, respuestas de Flows, etc.).
  • 6
    Crear y publicar el Flow Vía WhatsApp Manager (UI visual) o vía API programática con el Flow JSON.
  • 7
    Enviar el Flow via Template o CTA Los Flows se disparan desde un mensaje con botón Call-to-Action o dentro de una conversación activa.

05 / Flow JSON

Estructura del Flow JSON

Todos los Flows se definen mediante un documento JSON. Esta es la estructura base obligatoria:

FLOW JSON — ESTRUCTURA BASE 
{
  "version": "7.1",                  // Versión del spec (usar la más reciente)
  "data_api_version": "3.0",        // Solo si usas endpoint dinámico
  "routing_model": {
    "PANTALLA_1": ["PANTALLA_2"],   // Define navegación entre pantallas
    "PANTALLA_2": []              // Array vacío = pantalla terminal
  },
  "screens": [                      // Array de pantallas del flow
    {
      "id": "PANTALLA_1",           // ID único, MAYÚSCULAS, sin espacios
      "title": "Título visible",    // Aparece en el header del flow
      "terminal": false,           // true = pantalla final del flow
      "data": {},                  // Datos iniciales (inyectados por tu server)
      "layout": {
        "type": "SingleColumnLayout",
        "children": [              // Componentes de UI en esta pantalla
          // ... componentes aquí
        ]
      }
    }
  ]
}
ℹ️

IDs de pantalla: deben ser en MAYÚSCULAS, snake_case sin espacios. Ejemplo: BIENVENIDA, SELECCION_SERVICIO, CONFIRMACION_CITA.

Calendario con Slots Dinámicos (Agendamiento Real)

Para mostrar horarios disponibles según la fecha elegida, necesitas combinar CalendarPicker + data_exchange + tu endpoint. El flujo es:

  • 1
    Pantalla 1: El usuario elige la fecha CalendarPicker con on-click-action: data_exchange. Al confirmar, WhatsApp llama tu endpoint enviando la fecha seleccionada.
  • 2
    Tu endpoint consulta disponibilidad Recibe la fecha, consulta tu base de datos o API de calendario (Google Calendar, Outlook, sistema propio) y retorna los slots disponibles.
  • 3
    Pantalla 2: El usuario elige el horario Tu respuesta inyecta los slots como data-source en un RadioButtonsGroup o Dropdown dinámico.
  • 4
    Pantalla 3: Confirmación Muestra resumen y registra la cita. Retorna con on-click-action: complete.

08 / APIs Dinámicas

Conectar APIs Externas al Flow

Para mostrar datos en tiempo real de tu sistema (CRM, inventario, calendario, etc.), necesitas implementar un Data Exchange Endpoint: una URL HTTPS que WhatsApp llamará en momentos específicos del Flow.

Cuándo se llama tu endpoint

TriggerAcción JSONQué recibe tu server
Al abrir el Flow (pantalla inicial) name: "INIT" El número del usuario, datos iniciales del flow_token
Al hacer clic en un botón con data_exchange name: "data_exchange" Pantalla actual, datos del formulario ingresados hasta ese momento
Al cerrar el Flow (cancelar) name: "BACK" Señal de cancelación

Implementación del Endpoint (Node.js)

JAVASCRIPT — Express Endpoint 
const express = require('express');
const crypto = require('crypto');
const app = express();
app.use(express.json());

// Tu clave privada RSA para desencriptar
const PRIVATE_KEY = process.env.WA_PRIVATE_KEY;

app.post('/flows/endpoint', async (req, res) => {
  try {
    // 1. Desencriptar el payload de WhatsApp
    const decryptedBody = decryptPayload(req.body, PRIVATE_KEY);
    const { screen, data, flow_token, version, action } = decryptedBody;

    // 2. Manejar la acción INIT (primera carga)
    if (action === 'INIT') {
      const fechasBloqueadas = await obtenerFestivosYOcupados();
      return res.json(encryptResponse({
        screen: "ELEGIR_FECHA",
        data: {
          min_fecha: getTomorrow(),
          fechas_bloqueadas: fechasBloqueadas
        }
      }));
    }

    // 3. Manejar data_exchange de pantalla ELEGIR_FECHA
    if (screen === 'ELEGIR_FECHA' && action === 'data_exchange') {
      const fechaElegida = data.fecha; // "2025-09-15"
      const slots = await consultarSlotsDisponibles(fechaElegida);

      if (slots.length === 0) {
        // Sin disponibilidad, mostrar error
        return res.json(encryptResponse({
          screen: "ELEGIR_FECHA",
          data: { error_message: "No hay horarios para esa fecha. Elige otra." }
        }));
      }

      return res.json(encryptResponse({
        screen: "ELEGIR_HORA",
        data: {
          horarios_disponibles: slots.map(s => ({
            id: s.hora,
            title: s.etiqueta   // ej: "10:00 AM — Dr. García"
          })),
          fecha_seleccionada: fechaElegida
        }
      }));
    }

    // 4. Manejar BACK / cancelación
    if (action === 'BACK') {
      return res.json(encryptResponse({ screen: "ELEGIR_FECHA", data: {} }));
    }

  } catch (err) {
    console.error(err);
    res.status(500).json({ error: err.message });
  }
});

app.listen(3000);

09 / Seguridad

Encriptación de Datos (Flows Dinámicos)

Cuando un Flow tiene un endpoint dinámico, todos los datos se encriptan usando RSA-2048 + AES-128-GCM (encriptación híbrida). Debes generar un par de llaves y registrar la pública en Meta.

Generar el Par de Llaves

TERMINAL 
# Generar llave privada RSA 2048-bit
openssl genrsa -out private_key.pem 2048

# Extraer la llave pública
openssl rsa -in private_key.pem -pubout -out public_key.pem

# Ver el contenido (lo necesitarás para registrar en Meta)
cat public_key.pem

Registrar la Llave Pública en Meta

cURL — Registrar llave pública 
curl -X POST \
  https://graph.facebook.com/v20.0/{WABA_ID}/whatsapp_business_encryption \
  -H "Authorization: Bearer {SYSTEM_USER_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "business_public_key": "-----BEGIN PUBLIC KEY-----\nMIIBIjANBgkqh...\n-----END PUBLIC KEY-----"
  }'

Lógica de Desencriptación (Node.js)

JAVASCRIPT — Desencriptar payload 
const crypto = require('crypto');

function decryptPayload(body, privateKeyPem) {
  const {
    encrypted_aes_key,
    encrypted_flow_data,
    initial_vector
  } = body;

  // 1. Desencriptar la llave AES usando tu RSA privada
  const aesKey = crypto.privateDecrypt(
    {
      key: privateKeyPem,
      padding: crypto.constants.RSA_PKCS1_OAEP_PADDING,
      oaepHash: "sha256"
    },
    Buffer.from(encrypted_aes_key, 'base64')
  );

  // 2. Desencriptar los datos del flow con AES-128-GCM
  const iv = Buffer.from(initial_vector, 'base64');
  const TAG_LENGTH = 16;
  const encryptedData = Buffer.from(encrypted_flow_data, 'base64');
  
  const encryptedBody = encryptedData.subarray(0, -TAG_LENGTH);
  const authTag = encryptedData.subarray(-TAG_LENGTH);

  const decipher = crypto.createDecipheriv('aes-128-gcm', aesKey, iv);
  decipher.setAuthTag(authTag);

  const decrypted = Buffer.concat([
    decipher.update(encryptedBody),
    decipher.final()
  ]);

  return JSON.parse(decrypted.toString('utf-8'));
}
🔴

Seguridad crítica: Nunca expongas tu llave privada en el código fuente ni en el repositorio. Usa variables de entorno (process.env.WA_PRIVATE_KEY) o un servicio de secretos como AWS Secrets Manager, HashiCorp Vault o Google Secret Manager.

flow_ctaTexto del botón. Máximo 20 caracteres. Ej: "Agendar ahora".
flow_actionnavigate (abre en pantalla específica) o data_exchange (llama endpoint antes de mostrar).
flow_action_payload.screenID de la pantalla inicial que se mostrará al abrir el Flow.
flow_action_payload.dataDatos iniciales a inyectar en el Flow (ej: nombre del usuario, ID de cliente, etc.).

Método 2: Via Template Message (business-initiated)

ℹ️

Para enviar el Flow de forma proactiva (sin que el usuario haya iniciado conversación), debes crear un Template Message con un botón de tipo FLOW y aprobarlo en Meta. El template debe ser de categoría UTILITY o MARKETING. Los templates pueden tardar hasta 24h en ser aprobados.

Procesar la Respuesta (Node.js)

JAVASCRIPT — Procesar webhook 
app.post('/webhook', (req, res) => {
  res.sendStatus(200); // Responder SIEMPRE 200 inmediatamente

  const message = req.body?.entry?.[0]?.changes?.[0]?.value?.messages?.[0];
  
  if (message?.type === 'interactive' &&
      message.interactive?.type === 'nfm_reply') {
    
    const flowToken = message.interactive.nfm_reply.flow_token;
    const flowData = JSON.parse(message.interactive.nfm_reply.response_json);
    const userPhone = message.from;

    // ✅ Aquí procesas la respuesta del flow
    procesarRespuestaFlow({
      token: flowToken,
      datos: flowData,    // { nombre, fecha, horario, ... }
      telefono: userPhone
    });
  }
});

async function procesarRespuestaFlow({ token, datos, telefono }) {
  // Guardar en DB, notificar equipo, crear cita en CRM, etc.
  await db.guardarCita(datos);
  await enviarConfirmacion(telefono, datos);
}
⚠️

El webhook de Meta tiene un timeout de 20 segundos. Si no respondes con HTTP 200 a tiempo, Meta reintentará el envío. Siempre responde 200 inmediatamente y procesa la lógica de negocio de forma asíncrona (usando una queue o proceso en background).

JAVASCRIPT — Endpoint: construir slots por período del día 
if (screen === 'DIAS_DISPONIBLES' && action === 'data_exchange') {
  const fechaElegida = data.fecha; // "2026-04-16"

  // Consultar slots disponibles en tu sistema (CRM, Google Calendar, BD, etc.)
  const todosLosSlots = await obtenerSlots(fechaElegida, gestorId);

  // Separar por período
  const slotsMañana = todosLosSlots
    .filter(s => s.hora < '12:00')
    .map(s => ({
      id: s.hora,          // "08:00" — será el valor enviado al completar
      title: s.hora,       // "08:00" — texto del chip visible
      enabled: s.disponible // false = chip aparece gris/deshabilitado
    }));

  const slotsTarde = todosLosSlots
    .filter(s => s.hora >= '12:00')
    .map(s => ({
      id: s.hora,
      title: s.hora,
      enabled: s.disponible
    }));

  // Formatear la fecha para mostrar en UI: "Sábado 16 de abril"
  const fechaLabel = formatearFecha(fechaElegida, 'es-MX');

  return res.json(encryptResponse({
    screen: "HORAS_DISPONIBLES",
    data: {
      fecha_seleccionada: fechaElegida,
      fecha_label: fechaLabel,
      slots_manana: slotsMañana,
      slots_tarde: slotsTarde
    }
  }));
}

// Helper para formatear fecha
function formatearFecha(dateStr, locale) {
  const date = new Date(dateStr + 'T00:00:00');
  return date.toLocaleDateString(locale, {
    weekday: 'long', day: 'numeric', month: 'long'
  }); // → "sábado, 16 de abril"
}
⚠️

Slots deshabilitados (color gris): Cuando "enabled": false en un item del data-source, WhatsApp renderiza el chip en gris y el usuario no puede seleccionarlo. Úsalo para mostrar horarios ocupados de referencia. Si prefieres solo mostrar los disponibles, simplemente no incluyas los ocupados en el array.

Puntos Críticos y Errores Comunes

🔴

Los arrays dinámicos necesitan __example__: Todo campo en "data" de una pantalla debe incluir "__example__" con datos representativos. Sin esto, el Flow falla la validación al publicarlo, aunque los datos reales vengan del endpoint.

🔴

El name de los inputs debe ser único en todo el Flow: Si tienes hora_elegida en la sección mañana y hora_elegida en tarde, uno pisará al otro. Usa nombres distintos y en el endpoint decide cuál tiene valor.

⚠️

Terminal obligatorio: Exactamente una pantalla debe tener "terminal": true y su routing_model debe tener un array vacío []. Si el routing_model lista pantallas hijas, no puede ser terminal.

⚠️

Slots deshabilitados vs. no incluirlos: Mostrar chips grises (enabled: false) da contexto visual al usuario sobre horarios del día que "ya están ocupados". Si el horario ni siquiera existe en tu sistema, mejor no incluirlo en el array para no confundir.

✅

Testing con Flow en modo Draft: Antes de publicar el Flow, puedes enviarlo en modo "flow_mode": "draft" en el mensaje. Solo tú (el admin) podrás verlo. Úsalo para probar el endpoint y la navegación completa antes de que llegue a usuarios reales.
